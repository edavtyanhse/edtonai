## 🔴 КРИТИЧЕСКИЕ БАГИ (мешают пользователям)

### BUG-01 — Парсинг резюме запускается сразу при загрузке файла
**Проблема:** При drag-drop файла (`Step1Resume.tsx`) парсинг отправляется на бэк без подтверждения пользователя.
**Ожидание:** Файл загружается → пользователь видит превью текста → нажимает «Далее» / «Разобрать» → только тогда вызывается `/v1/resumes/parse`.
**Файлы:** `frontend/src/pages/wizard/Step1Resume.tsx`, `frontend/src/lib/file-processing.ts`

### BUG-02 — Нельзя вернуться назад и загрузить другую вакансию
**Проблема:** После ввода вакансии в `Step2Vacancy.tsx` нет явного способа сбросить данные и загрузить другую.
**Ожидание:** Кнопка «Изменить вакансию» / «Загрузить другую» сбрасывает `parsedVacancy` в WizardContext и возвращает шаг 2 в режим ввода.
**Файлы:** `frontend/src/pages/wizard/Step2Vacancy.tsx`, `frontend/src/context/WizardContext.tsx`

### BUG-03 — Скор непредсказуем: после улучшения может не вырасти
**Проблема (корневая причина):**
- LLM недетерминирован → один и тот же резюме при повторном анализе даёт разный скор
- При адаптации бэкенд не сообщает анализатору, какие навыки были **намеренно добавлены**
- Итог: пользователь видит «до: 65 / после: 62» и теряет доверие к продукту
- Критичный кейс: навык был в «missing skills» → пользователь выбрал его для улучшения → он попал в резюме → повторный анализ снова показывает его как missing

**Выбранный подход: LLM с контекстом применённых улучшений**

#### Этап 1 — Адаптация (уже частично есть, нужно усилить)

В промпт `GENERATE_UPDATED_RESUME_PROMPT` добавить явное требование:
- Каждый выбранный `checkbox_id` ДОЛЖЕН быть материализован в тексте резюме
- Если checkbox = навык → добавить в раздел Skills/Experience
- Если checkbox = достижение → добавить конкретную формулировку
- LLM возвращает `applied_checkbox_ids` — список того, что реально добавлено (уже есть)

#### Этап 2 — Контекстный re-анализ (новое)

Новый метод `MatchService.analyze_with_context()`:
```python
async def analyze_with_context(
    self,
    parsed_resume: dict,
    parsed_vacancy: dict,
    original_analysis: dict,         # анализ до адаптации
    applied_checkbox_ids: list[str],  # что было выбрано и добавлено
    applied_improvements: list[dict], # детали каждого улучшения
) -> MatchAnalysisResult: ...
```

Промпт для контекстного анализа говорит LLM:
> «Ты уже анализировал это резюме ранее. Вот оригинальный анализ. С тех пор пользователь намеренно добавил следующие навыки/улучшения: [...]. Они присутствуют в тексте резюме. НЕ помечай их как отсутствующие. Пересчитай скор с учётом этих добавлений.»

Cache key для контекстного анализа включает `hash(applied_checkbox_ids)` → отдельная запись в кеше.

#### Этап 3 — Верификация (новое)

После адаптации — быстрая проверка: реально ли добавленные навыки присутствуют в тексте.
Простая `set intersection` между ключевыми словами улучшений и словами адаптированного резюме (без LLM, детерминировано).
Если навык не найден → `partially_applied: true` → предупреждение пользователю.

#### Этап 4 — Фронт

- После адаптации автоматически запускать контекстный re-анализ
- Показывать `Скор до → Скор после` с зелёной дельтой
- В списке улучшений (Step 3) навыки из `applied_checkbox_ids` помечать как ✓ закрытые
- Если `partially_applied` — жёлтый ⚠️ с подсказкой

**Файлы к изменению:**
- `backend/services/match.py` — новый метод `analyze_with_context()`
- `backend/integration/ai/prompts.py` — новый промпт `MATCH_WITH_CONTEXT_PROMPT`
- `backend/services/adapt.py` — после адаптации вызывать контекстный анализ
- `backend/schemas/responses/` — добавить `context_analysis` в `AdaptResumeResponse`
- `frontend/src/pages/wizard/Step4Improvement.tsx` — показывать before/after скор
- `frontend/src/pages/wizard/Step3Analysis.tsx` — помечать закрытые навыки

## 🟠 АУТЕНТИФИКАЦИЯ — Замена Supabase

> Supabase используется **только для аутентификации** (не для хранения данных).
> Данные хранятся в собственной PostgreSQL.

### AUTH-01 — Убрать Supabase, поднять собственный auth-сервис

**Архитектурное решение:**
- Новый модуль `backend/auth/` — собственный JWT-сервис
- Таблицы: `users`, `oauth_accounts`, `email_verifications`, `refresh_tokens`
- Access token: JWT 15 мин, Refresh token: 30 дней (httpOnly cookie)
- Миграция через Alembic

**Провайдеры OAuth:**
- **Google** — OAuth 2.0 (приоритет #1, самый популярный)
- **VK** — OAuth 2.0 (`api.vk.com/oauth/authorize`)
- **Яндекс** — OAuth 2.0 (`oauth.yandex.ru/authorize`)
- **Email/пароль** — bcrypt hash, верификация через письмо

**Файлы к созданию:**
```
backend/auth/
├── router.py          # /auth/login, /auth/register, /auth/oauth/*, /auth/refresh, /auth/logout
├── service.py         # логика входа, регистрации, верификации
├── oauth/
│   ├── google.py
│   ├── vk.py
│   └── yandex.py
├── jwt.py             # create/verify access + refresh tokens
├── password.py        # bcrypt utils
└── models.py          # User, OAuthAccount, RefreshToken ORM
```

**Фронт:**
- Убрать `frontend/src/lib/supabase.ts`
- Переписать `frontend/src/context/AuthContext.tsx` на собственный API
- Новые страницы: LoginPage, RegisterPage, EmailVerificationPage
- OAuth redirect flow (popup или redirect)

### AUTH-02 — Email-рассылка с собственного домена

**Что нужно:**

```
backend/integration/email/

```

## 🟡 ДИЗАЙН И UX — Ориентир: Jobright.ai

> Jobright фокус: анализ резюме + поиск вакансий.
> Наш фокус: **адаптация резюме под конкретную вакансию**.
> Берём у Jobright: чистый SaaS-дизайн, dashboard-подход, чёткие метрики.

### UX-01 — Редизайн главной страницы (Dashboard)

**Вместо "выбери режим":**
- Верхняя панель с последними 3 адаптациями
- Кнопка «Новая адаптация» → запускает wizard
- Блок «Мои резюме» (сохранённые базовые резюме)
- Блок «История» с badge'ами скора (было → стало)

### UX-02 — Редизайн Wizard

**Шаг 1 — Резюме:**
- Drag-drop зона + кнопка «Вставить текст»
- После загрузки → превью текста + кнопка «Продолжить» (парсинг НЕ запускается до нажатия)
- Индикатор прогресса: «Анализируем резюме...»

**Шаг 2 — Вакансия:**
- Два таба: «Ссылка» / «Текст»
- Кнопка «← Назад» работает без потери данных
- Кнопка «Изменить вакансию» после ввода
- Preview карточки вакансии (название, компания, ключевые требования)

**Шаг 3 — Анализ:**
- Score ring (большой, по центру) с анимацией заполнения
- Breakdown по категориям: Hard Skills / Soft Skills / Опыт / Образование
- Список «Что совпадает ✓» и «Чего не хватает ✗»
- Кнопка «Улучшить резюме →»

**Шаг 4 — Улучшение:**
- Чеклист группируется по категориям (не плоский список)
- Каждый пункт показывает: что добавить + почему важно для этой вакансии
- После адаптации: сравнение «было / стало» + новый скор
- Скор-дельта выделяется зелёным: «+12 баллов»

### UX-03 — Компонент Score (переиспользуемый)

Создать `ScoreRing.tsx`:
- SVG-кольцо с процентом
- Цвет: красный (0-40) / жёлтый (40-70) / зелёный (70-100)
- Анимация при первом рендере
- Мини-версия для карточек истории

### UX-04 — История версий

- Timeline вместо таблицы
- Каждая версия: дата, вакансия, скор-до, скор-после, применённые улучшения
- Кнопки: «Скачать PDF», «Сравнить», «Использовать как базу»
