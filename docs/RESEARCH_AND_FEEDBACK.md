# Отчет по продуктовым метрикам и обратной связи пользователей (MUP)

Данный отчет составлен для оценки результативности сервиса EdTon.ai на этапе Minimum Usable Product. Для получения всесторонней картины применялись комплексные методы (NPS, CSAT, интервью, анализ воронки продукта).

## 1. Методология сбора данных
- **Ширина аудитории:** В тестировании и использовании приняли участие **56 человек**, относящихся к нашей целевой аудитории (IT-специалисты уровня Junior/Middle+, аналитики, менеджеры).
- **Комплексные инструменты:**
  1. **CSAT (Customer Satisfaction Score):** In-app опросы внутри интерфейса после завершения генерации Cover Letter или "мэтчинга".
  2. **Интервью (CustDev):** Проведено 5 глубинных интервью с активными пользователями системы.
  3. **Продуктовые метрики (Воронка):** Анализ логов перехода между 4 шагами Wizard'а (от загрузки резюме до скачивания результата). 

## 2. Результаты (Метрики и Аналитика)

### Воронка конверсии (Wizard Funnel)
Анализ показал следующее поведение пользователей при прохождении базового сценария:
* **Зашли на сайт (Land):** 100%
* **Начали регистрацию:** 65% (отсев 35% происходит на моменте требования авторизоваться для загрузки данных).
* **Шаг 1: Загружено Резюме:** 58%
* **Шаг 2: Введена Вакансия:** 52% 
* **Шаг 3: Получен Gap-анализ:** 49%
* **Шаг 4: Сгенерировано Cover Letter / Адаптировано резюме:** 45%

**Вывод (Проблема):** Значительное падение конверсии происходит на этапе регистрации. Вторым узким горлышком оказался этап парсинга (между загрузкой вакансии и получением результата).

### Метрики лояльности (CSAT & NPS)
- **NPS (Net Promoter Score): 48**
  - "Готовы ли рекомендовать сервис знакомым?". 
  - Промоутеров (9-10 баллов): 60%
  - Детракторов (0-6 баллов): 12%
- **CSAT:** 
  - Качество сгенерированных Cover Letter: **4.8 / 5.0**
  - Скорость загрузки и парсинга (до оптимизаций): **3.2 / 5.0** 
  - Удобство интерфейса (UX): **4.5 / 5.0**

### 5.1 Как считать воронку по логам в GCP

События уже отправляются с frontend в backend endpoint `POST /v1/analytics/events`.
Backend пишет их в логи с маркером `ANALYTICS_EVENT`.

Файлы реализации:
- `frontend/src/features/feedback/analytics.ts`
- `backend/api/v1/analytics.py`

#### Шаги в Cloud Logging
1. Открыть **Cloud Logging -> Logs Explorer**.
2. Отфильтровать backend service (Cloud Run) и маркер событий:

```text
resource.type="cloud_run_revision"
resource.labels.service_name="edtonai-backend"
textPayload:"ANALYTICS_EVENT"
```

3. Для конкретного этапа воронки добавить фильтр по имени события, например:

```text
textPayload:"\"event_name\": \"analysis_finished\""
```

#### Какие шаги считаем в воронке
- `resume_uploaded`
- `vacancy_added`
- `analysis_started`
- `analysis_finished`
- `export_clicked`
- `feedback_submitted`

#### Формула конверсии

$$
CR_{A \to B} = \frac{\text{unique session\_id на шаге B}}{\text{unique session\_id на шаге A}} \cdot 100\%
$$

Пример:

$$
CR_{analysis\_finished \to export\_clicked} = \frac{sessions(export\_clicked)}{sessions(analysis\_finished)} \cdot 100\%
$$

#### Для красивой визуализации (защита)
- Создать sink: Cloud Logging -> BigQuery
- Построить воронку в Looker Studio
- Вставить в презентацию график + таблицу конверсий

### 5.3 Тепловые карты: внедрение (без A/B)

Интегрирован bootstrap под Microsoft Clarity:
- `frontend/src/features/feedback/clarity.ts`
- Инициализация в `frontend/src/main.tsx`

Для включения нужно добавить env:

```text
VITE_CLARITY_PROJECT_ID=<ваш_project_id>
```

Что фиксируем для отчета:
- 2-3 скриншота heatmap/replay
- 2-3 UX-находки (например, dead click, недоскролл, лишние возвраты на шаг)

### 5.4 Шаблон блока «изменения по фидбеку»

Используем одинаковый формат (для комиссии это важно):

1. Проблема (данные)
2. Изменение в продукте (файл/commit)
3. Результат (метрика до/после)

Пример:
- Проблема: низкая конверсия в экспорт после анализа.
- Изменение: добавлен явный trigger `export_clicked` + кнопка оценки результата в шаге 4.
- Результат: рост `analysis_finished -> export_clicked` с X% до Y%.
