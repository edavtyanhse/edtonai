## 1) Что именно требует критерий 10
Для оценки 10 по MUP по блоку фидбека нужны одновременно:
1. Широкая аудитория.
2. Комплексные методы: анкеты, NPS, CSAT, интервью, поведенческие метрики, тепловые карты, A/B.
3. Глубокий анализ с визуализацией.
4. Доказуемые изменения в продукте по результатам фидбека.

## 2) Разделение на «обязательно до защиты» и «допустимо как план»

### 2.1 Обязательно реализовать до защиты (в коде)
- In-app CSAT (оценка 1-5 + комментарий).
- In-app NPS (оценка 0-10 + комментарий).
- Сохранение в БД: оценка, тип метрики (CSAT/NPS), контекст шага, версия UI, user segment, timestamp.
- Точка показа:
  - CSAT: после получения результата адаптации.
  - NPS: после завершения полного сценария (скачивание/экспорт/финальный шаг).
- События воронки (behavior events):
  - step_entered, resume_uploaded, vacancy_added, analysis_started, analysis_finished, export_clicked, feedback_submitted.

### 2.2 Допустимо как частично реализованный/этапный пункт
- Тепловые карты: подключить внешнюю аналитику (Microsoft Clarity или PostHog) + показать скриншоты/дашборд.
- A/B тест: реализовать минимально как feature flag A/B (variant A/B в событии), сравнение конверсии/CSAT.
- Интервью: для демо-стенда допускается синтетический набор 8 интервью (сегменты, боли, цитаты, выводы), с явной пометкой «учебная выборка».

## 4) Как закрыть каждый пункт критерия (доказательная матрица)

1. Широкая аудитория
- Минимум 30-50 респондентов из 3 сегментов.
- В отчете: таблица сегментов и размеров выборки.

2. NPS
- Реализуется in-app шкалой 0-10.
- Формула: NPS = %Promoters(9-10) - %Detractors(0-6).
- В отчете: распределение по шкале + итоговый NPS.

3. CSAT
- Реализуется in-app оценкой 1-5 после ключевого результата.
- В отчете: среднее значение, медиана, динамика по неделям.

4. Интервью
- 5-8 интервью с целевой аудиторией.
- В отчете: топ-5 болей + цитаты + решения.

5. Метрики поведения
- Воронка по шагам (upload -> analysis -> export).
- В отчете: конверсия каждого шага и причины отвалов.

6. Тепловые карты
- Скриншоты кликов/скролла по экрану результата.
- В отчете: 2-3 находки (где пользователи «теряются»).
- Для текущего дедлайна: приоритет Microsoft Clarity (самая быстрая интеграция, бесплатно, веб).

7. A/B тест
- A: текущий CTA, B: переработанный CTA.
- Метрики сравнения: CTR, completion rate, CSAT.

8. Изменения в продукте по фидбеку
- Формат доказательства в отчете:
  - Проблема (данные)
  - Изменение (ссылка на PR/коммит)
  - Результат (метрика до/после)

## 8) Уточнение по вашим ограничениям (быстрый защитный вариант)

### 8.1 Интервью: «нагенерируем» корректно для защиты
- Делаем 8 синтетических интервью: 3 junior, 3 middle, 2 career-switcher.
- Для каждого: контекст, боль, цитата, приоритет проблемы, предлагаемое изменение.
- В отчете обязательно пометка: «данные для учебного демо, методика приближена к реальной».
- На защите формулировка: «пилотный качественный этап, в проде заменяется на реальные интервью». 

### 8.2 Поведенческая аналитика из логов (так как логи уже отливаются)
События, которые должны логироваться в structured JSON:
- `step_entered`
- `resume_uploaded`
- `vacancy_added`
- `analysis_started`
- `analysis_finished`
- `export_clicked`
- `feedback_submitted`

Обязательные поля в логах:
- `event_name`
- `user_id` (или анонимный hash)
- `session_id`
- `timestamp`
- `step`
- `ui_variant` (A/B)
- `user_segment`

Как искать в Google Cloud Logging:
- Фильтр по сервису backend + `jsonPayload.event_name:*`.
- Отдельно по этапам воронки: `jsonPayload.event_name="analysis_finished"` и т.д.
- Конверсию считать как отношение уникальных `session_id` между шагами.
- Для отчета выгрузить в BigQuery (sink) и построить графики в Looker Studio.

### 8.3 Тепловые карты: насколько сложно
- Сложность: низкая/средняя.
- Время интеграции: 1-3 часа для базового запуска (добавить JS snippet, проверить запись сессий).
- Что дают: клики, скролл, rage-click, dead-click, session replay.
- Ограничение: не заменяют продуктовые события и воронку, а дополняют их.

### 8.4 Простая гипотеза для A/B (готово для защиты)
Гипотеза:
- Если изменить CTA после результата анализа с нейтрального «Продолжить» на ценностный «Скачать адаптированное резюме»,
- то вырастет completion rate финального шага и CSAT.

Дизайн:
- Вариант A: `Продолжить`.
- Вариант B: `Скачать адаптированное резюме` + подпись `PDF за 1 клик`.
- Метрики: `export_clicked`/`analysis_finished` (completion), CSAT, время до клика.
- Минимальная выборка для демонстрации: 10-20 пользователей (пилот).

### 8.5 Что из этого есть «нативно» в GCP
Есть в GCP:
- Cloud Logging (сбор и фильтрация событий)
- Cloud Monitoring + Alerting
- BigQuery (хранилище событий)
- Looker Studio (дашборды)

Нет «из коробки» как готовая heatmap-система:
- Тепловые карты и replay лучше брать внешними: Microsoft Clarity / PostHog / Hotjar.

Практичный стек для вашей защиты:
- Логи событий: backend -> Cloud Logging
- Аналитика/воронка: Cloud Logging sink -> BigQuery -> Looker Studio
- Тепловые карты: Microsoft Clarity
