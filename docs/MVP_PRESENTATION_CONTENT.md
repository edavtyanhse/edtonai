## Слайд 2: Проблема и Решение
**Проблема:**
- Рекрутеры тратят ~6 секунд на резюме.
- ATS (роботы) отклоняют 75% резюме из-за отсутствия ключевых слов.
- Кандидаты тратят часы на ручную адаптацию под каждую вакансию.

**Решение:**
- **EdTon.ai**: Адаптация резюме за 3 клика.
- AI анализирует вакансию, находит разрывы (gaps) и сам переписывает текст, повышая шансы на интервью.

## Слайд 4: Архитектура (C4 Context & Container)
*(Вставьте диаграммы C4 L1 и L2 из `docs/ARCHITECTURE.md`. Можно на одном слайде или разбить на два)*

**1. Context (L1) — Взаимодействие с внешним миром**
- **Пользователь (Job Seeker)**: Инициирует адаптацию, загружает файлы, получает результат.
- **EdTon.ai System**: Центральный хаб. Принимает запросы, управляет процессом, но не хранит чувствительные данные авторизации (делегирует Supabase).
- **Supabase (Auth/DB)**: Внешний BaaS. Отвечает за аутентификацию (JWT) и надежное хранение данных (PostgreSQL с RLS политиками).
- **AI Providers (Brains)**:
    - *DeepSeek V3*: "Мозг" системы. Выполняет сложный анализ (Gap Analysis) и креативное переписывание текста.
    - *Groq (Llama 3)*: "Глаза" системы. Сверхбыстрый парсинг текста из PDF и веб-страниц вакансий.

**2. Container (L2) — Внутреннее устройство**
- **SPA (Frontend Container)**:
    - *Tech*: React, TypeScript, Vite.
    - *Role*: Отвечает за интерактивность (Wizard), валидацию форм и отображение Diff-ов. Общается с API по REST.
- **REST API (Backend Container)**:
    - *Tech*: Python, FastAPI, Pydantic.
    - *Role*: Оркестратор. Принимает "сырые" данные, чистит их, отправляет промпты в LLM, сохраняет результат в БД.
- **Database (Data Container)**:
    - *Tech*: PostgreSQL.
    - *Role*: Хранит JSON-структуры резюме (до и после), тексты вакансий и историю изменений.

## Слайд 7: Развертывание (Deployment)
*(Вставьте Deployment диаграмму из `docs/ARCHITECTURE.md`)*

**Инфраструктура (Google Cloud Platform)**
- **Cloud Run (Serverless)**:
    - Контейнеры запускаются только при поступлении запроса (Scale to Zero).
    - Это экономит бюджет в простое и автоматически масштабируется при нагрузке.
- **Google Cloud Build (CI/CD)**:
    - Пайплайн полностью автоматизирован.
    - *Push to main* -> *Linter Check (ESLint, Pylint)* -> *Build Docker* -> *Deploy to Cloud Run*.
- **Code Quality Gates**:
    - В CI встроены жесткие проверки стиля кода.
    - **Frontend**: ESLint + Prettier (Quality + Formatting).
    - **Backend**: Ruff (Linter) + Black (Formatter) + Pytest (Tests).
    - Если линтер находит ошибку, деплой отменяется автоматически.
- **External Services Strategy**:
    - Мы не хостим БД и AI-модели сами, а используем Managed Services (Supabase, DeepSeek API), чтобы сосредоточиться на бизнес-логике, а не на администрировании серверов.
