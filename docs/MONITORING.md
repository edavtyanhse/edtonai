# Стратегия Мониторинга и Алертинга

В данном документе описывается подход к мониторингу работы приложения EdtonAI, сбору метрик и реагированию на инциденты (алертинг), что является важной частью обеспечения нефункциональных требований (NFR) к производительности и надежности уровня Production.

## 1. Метрики и Дашборды (Grafana + Prometheus)

В проекте используется связка **Prometheus** (для сбора и агрегации метрик) и **Grafana** (для визуализации).  
Файлы конфигураций дашбордов находятся в репозитории: `monitoring/edtonai_dashboard.json`.

**Ключевые отслеживаемые метрики:**
- **RPS (Requests Per Second):** Количество запросов к API FastAPI в секунду.
- **Latency (Задержка):** Время выполнения запросов. Отдельное внимание уделяется AI-эндпоинтам (генерация писем, матчинг), так как эти операции могут занимать от 5 до 20 секунд.
- **Error Rate (Процент ошибок):** Количество `5xx` и `4xx` HTTP-статусов, собираемых с API.
- **Ресурсы контейнеров (CPU / RAM):** Метрики, собираемые через `cAdvisor` с Docker-контейнеров, для оценки нагрузки на узлы балансировщика, базы данных и воркеров приложения.
- **Database Connections:** Количество активных сессий к PostgreSQL (Supabase) и пул соединений `SQLAlchemy`.

## 2. Health Checks (Проверки работоспособности)

Приложение реализует эндпоинт `/health`, доступный по корневому адресу API. Узел активно опрашивается балансировщиком нагрузки:

- Проверка доступности самого процесса FastAPI (возврат HTTP 200).
- Проверка доступности базы данных (выполняется тестовый запрос `SELECT 1` с помощью `SQLAlchemy AsyncSession`).  
Если БД недоступна, статус меняется на `degraded`, что позволяет оркестратору временно исключить узел из ротации.

## 3. Логирование приложения (Application Logging)

- Логи собираются в stdout/stderr стандартным модулем `logging` в формате JSON или структурированного текста (через Gunicorn/Uvicorn access logs). 
- Структура логов включает: `timestamp`, `level` (INFO, WARNING, ERROR), `module`, и контекст запроса.
- В критичных бизнес-узлах (провайдеры LLM) все ошибки парсинга, таймауты и отказы LLM логируются с traceback-информацией.

### 3.1 Продуктовые события воронки (MUP Feedback)

Для MUP добавлен endpoint `POST /v1/analytics/events`, который пишет события в логи с маркером `ANALYTICS_EVENT`.

События:
- `step_entered`
- `resume_uploaded`
- `vacancy_added`
- `analysis_started`
- `analysis_finished`
- `export_clicked`
- `feedback_submitted`

Ключевые поля события:
- `event_name`
- `session_id`
- `step`
- `ui_variant`
- `user_segment`
- `occurred_at`

Пример фильтра в Cloud Logging:

```text
resource.type="cloud_run_revision"
resource.labels.service_name="edtonai-backend"
textPayload:"ANALYTICS_EVENT"
```

Пример фильтра по шагу:

```text
textPayload:"\"event_name\": \"analysis_finished\""
```

Конверсию шагов считаем по уникальным `session_id`:

$$
CR_{A \to B}=\frac{sessions(B)}{sessions(A)}\cdot 100\%
$$

## 4. Стратегия Алертинга

План по внедрению системы оповещения строится на **Prometheus Alertmanager**:
1. **Critical Alerts:** Отправляются в Telegram-канал разработчиков. 
   *Триггеры:* Доступность `/health` упала; количество `5xx` ошибок превышает 5% за 1 минуту; база данных недоступна более 30 секунд.
2. **Warning Alerts:** Отправляются на Email.
   *Триггеры:* Высокая утилизация RAM (>85%) на сервере приложения; долгий ответ от стороннего API (LLM).

## 5. Безопасность и NFR (Non-Functional Requirements)

- **Надежность:** Обеспечивается контейнеризацией всех узлов (Docker-compose) и автоматическим перезапуском (`restart: always`).
- **Безопасность API:** 
  - Валидация всех входящих данных осуществляется через схемы `Pydantic`.
  - При возникновении ошибки клиенту отдаются только безопасные детали (без экспонирования системных стеков вызовов или SQL-запросов), реализовано с помощью кастомных Exception Handlers (`errors/handlers.py`).
  - **Авторизация/Аутентификация:** Модуль `backend/auth` обеспечивает JWT авторизацию и OAuth2 для безопасной работы с персональными данными пользователей (резюме, профили). Пароли хэшируются алгоритмом bcrypt.
